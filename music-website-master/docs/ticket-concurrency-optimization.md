# 高并发抢票系统 — 性能优化对比报告

**测试日期**：2026-06-27
**测试环境**：单机 Spring Boot + MySQL + Redis，JMeter 本地压测
**测试场景**：200 人同时抢 100 张演唱会门票

---

## 一、采用的技术方案

| 技术 | 解决的问题 | 实现方式 |
|------|-----------|---------|
| **Redis Lua 原子扣库存** | 超卖 | GET + 检查 + DECR 在一条 Lua 脚本中原子执行 |
| **SETNX 用户去重** | 一人多抢 | `SET key NX EX` 幂等标记，先卡位再扣钱 |
| **@Async 异步写 MySQL** | 响应阻塞 | 订单写入独立线程池，主线程 ~5ms 返回 |
| **MySQL 原子 UPDATE 扣余额** | 余额超扣 | `UPDATE SET yinbi = yinbi - ? WHERE yinbi >= ?` |
| **SETNX 先于扣款** | 重复扣款 | 先占位再扣钱，扣钱失败释放占位 |
| **定时 Stock 对账** | 最终一致性 | 每 5 分钟 MySQL 实际销量校正 Redis 库存 |

## 三、传统方案为什么慢 / 为什么不可靠

### 纯 MySQL 方案的问题

```sql
-- 传统做法：SELECT FOR UPDATE 悲观锁
SELECT stock FROM ticket_tier WHERE id = ? FOR UPDATE;
-- 检查 stock > 0
UPDATE ticket_tier SET stock = stock - 1 WHERE id = ?;
COMMIT;
```

| 问题 | 影响 |
|------|------|
| 行锁排队 | 200 个请求串行等待同一行的锁，响应时间线 \[50, 100, 150, 200...\] ms |
| 死锁风险 | 扣余额 + 扣库存两个表交叉加锁 |
| DB 连接耗尽 | 每个等待锁的请求占用一个 DB 连接 |
| 无法水平扩展 | 锁在 DB 层，加机器不解决问题 |

### 本项目方案

```
请求 → SETNX(Redis,1ms) → Lua DECR(Redis,1ms) → 响应(5ms)
                                                    ↓
                                            @Async → MySQL INSERT(后台)
```

| 优势 | 原理 |
|------|------|
| Redis 单线程 + Lua 原子 | 无锁竞争，O(1) 完成 |
| 异步写 MySQL | 不阻塞用户响应，DB IO 对用户透明 |
| 连接池压力小 | 只有异步线程池消费 DB 连接 |
| 可水平扩展 | Redis 无状态竞争，加应用实例即可提升 QPS |
